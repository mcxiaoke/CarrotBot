# 🤖 OmniBot-Service (统一企业消息网关) 实施计划书

## 1. 项目背景与目标
构建一个高扩展性、模块化的 Node.js 消息网关服务。
- **阶段一目标**: 实现企业微信应用消息的接收与回复，支持内部系统的 API 主动推送。
- **阶段二目标**: 验证架构扩展性，低成本接入 Telegram 等其他 IM 平台。
- **架构要求**: TypeScript 强类型、基于适配器模式解耦 IM 平台、基于命令模式解耦业务动作。

---

## 2. 技术栈基准字典
- **Runtime**: Node.js (v20 LTS)
- **Language**: TypeScript (v5.x)
- **Web Framework**: Fastify (极致性能与极简路由)
- **Validation**: Zod (用于 API 负载及环境变量的严格校验)
- **HTTP Client**: Axios 或原生 Fetch (用于调用企业微信/TG API)
- **Logging**: Pino (高性能结构化日志)
- **Testing**: Vitest (兼容 Jest 且速度更快)
- **Deployment**: Docker + Docker Compose (或 k8s)

---

## 3. 详细实施路线图 (预计周期: 4-5 周)

### 阶段一：基础设施与底层核心设计 (Week 1)
**目标**：搭建工程骨架，定义全局核心接口。

- [ ] **1.1 初始化工程**：配置 `tsconfig.json`, `ESLint`, `Prettier`, `lint-staged` 等工程化规范。
- [ ] **1.2 引入 Web 框架**：搭建 Fastify 基础实例，配置全局错误捕获、404 处理及基础日志中间件。
- [ ] **1.3 定义核心抽象 (Core Interfaces)**：
  - 定义 `IAdapter` 接口 (必须包含 `parseRequest`, `sendMessage` 等方法)。
  - 定义 `StandardMessage` 接口 (系统内部统一流转的消息体结构)。
  - 定义 `IActionHandler` 接口 (用于处理具体业务指令)。
- [ ] **1.4 配置管理**：基于 Zod 编写 `config.ts`，严格校验从 `.env` 读取的企业微信 `CORP_ID`, `CORP_SECRET`, `TOKEN`, `ENCODING_AES_KEY` 等关键参数。

**🚩 里程碑 1**：项目骨架能够成功 `npm run dev` 运行，通过所有 Lint 检查。

### 阶段二：企业微信适配器与消息链路打通 (Week 2)
**目标**：攻克最核心的企业微信回调加解密机制，实现消息的“收与发”。

- [ ] **2.1 开发 WeCom 鉴权中间件**：处理企业微信 Webhook 接入时的 URL 验证（VerifyURL）流程。
- [ ] **2.2 开发 WeCom 加解密模块**：实现基于 `AES-256-CBC` 的微信特殊消息加解密算法（这是企业微信接入的**最大难点**）。
- [ ] **2.3 实现 `WeComAdapter`**：
  - 编写 `sendMessage` 方法，实现获取 `access_token` (需要加入内存/Redis缓存机制以免触发限流) 并调用发送接口。
  - 编写 `parseRequest` 方法，将解密后的 XML 转换为 `StandardMessage`。
- [ ] **2.4 Echo 测试**：实现最简单的路由，原样将用户发送的内容回复给用户。

**🚩 里程碑 2**：在企业微信后台成功配置回调 URL，可以在企业微信客户端与机器人完成基础对话交互。

### 阶段三：指令路由引擎与业务动作解耦 (Week 3)
**目标**：让机器人变“聪明”，能理解指令并分发。

- [ ] **3.1 实现 MessageRouter**：构建路由分发器，根据文本前缀（如 `/`）或正则模式匹配，将 `StandardMessage` 派发给对应的 Action。
- [ ] **3.2 实现基础 Actions (命令模式)**：
  - 开发 `PingAction`：回复 `/ping` 返回系统运行 uptime 等信息。
  - 开发 `HelpAction`：回复 `/help` 返回当前支持的指令列表。
- [ ] **3.3 上下文管理 (可选扩展)**：预留中间件机制，用于未来记录用户会话上下文（Session），支持连续对话。

**🚩 里程碑 3**：发送不同指令，机器人能正确路由并执行对应业务逻辑。

### 阶段四：主动推送 API 与第三方平台扩展 (Week 4)
**目标**：开放能力给内部系统，并验证架构设计的扩展性。

- [ ] **4.1 开发 Push API Endpoint**：
  - 实现 `POST /api/v1/push` 接口。
  - 使用 Zod 定义请求体校验规则 (平台类型、接收人、消息内容)。
- [ ] **4.2 Push API 安全机制**：实现简单的 API-Key / Bearer Token 校验，防止内部接口被恶意调用。
- [ ] **4.3 验证多平台扩展 (开发 `TelegramAdapter`)**：
  - 接入 Telegram Bot API。
  - 实现 Webhook 接收与消息发送，并直接接入现有的 `MessageRouter`，无需修改任何 Action 代码。

**🚩 里程碑 4**：可通过 Postman 或 cURL 请求 API 成功触发企业微信消息推送；TG 机器人能在同一套系统中完美运行相同指令。

### 阶段五：测试、部署与运维监控 (Week 5)
**目标**：保障上线质量，实现自动化部署。

- [ ] **5.1 单元测试**：使用 Vitest 为核心机制（Router 分发逻辑、AES 加解密类）编写单元测试，保证后续迭代的安全性。
- [ ] **5.2 容器化 (Docker)**：编写 `Dockerfile` (推荐使用 Node Alpine 或 Distroless 镜像减小体积)。并提供 `docker-compose.yml` 示例。
- [ ] **5.3 CI/CD (GitHub Actions / GitLab CI)**：配置自动化流程，提交代码自动运行 Lint 和测试。
- [ ] **5.4 完善文档**：编写完整的 README.md，包含环境变量说明、本地启动步骤、新增指令的开发规范。

## 4. 风险评估与应对策略
1. **企业微信 API 访问限流**：
   - *对策*：务必全局单例化管理并缓存 `access_token`（有效期 7200 秒），避免每次发消息都请求 Token。
2. **回调处理超时重试**：
   - *风险*：企业微信要求 5 秒内响应，否则会重试，可能导致同一业务指令被执行多次。
   - *对策*：网关层接收消息后，尽早返回 `HTTP 200` 给企微服务器。若业务处理耗时较长，应采用**异步执行+主动推送结果**的模式；系统层可加入 MsgId 去重机制。
